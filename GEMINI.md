プロジェクト AI エージェント ルールブック

これは、このリポジトリのコード修正およびレビューを行う AI エージェント（@gemini-cli）のための基本ルールとコンテキストです。

開発の目的と技術スタック

主な目的: このプロジェクトはウェブアプリケーション開発であり、ユーザー体験とセキュリティを最優先とします。

使用技術: Google Apps Script (GAS), JavaScript (ES6+), Tailwind CSS

コーディング規約: 変数名はキャメルケースを使用。

Issue 処理のルール（自律実行の基準）

AIエージェントは、以下の条件を満たした Issue のみを自動で処理し、Pull Request を作成します。

必須トリガー: Issueのラベルに ready-for-ai が付与されていること。

必須の記述要素: Issue本文には、以下の項目が明確に含まれていること。

ゴール: 達成すべき具体的な結果 (例: ログイン機能を修正)。

ターゲットファイル: (任意) 修正対象のファイルパスや関数名 (例: src/auth.jsのloginUser関数)。

テスト基準: (任意) 修正後にパスすべきテストの条件。

コード修正の基本戦略 (最重要)

コードを修正する際は、以下の戦略を優先すること。これにより、安全かつ確実にタスクを完了させる。

最優先事項: シンプルなツールを好むこと

ファイルの検索や書き換えには、複雑なシェルコマンド (run_shell_command) の使用を極力避けること。

代わりに、search_file_content や replace_in_file のような、より単純で安全なツールを優先して使用すること。これは、セキュリティエラーを回避するための最も重要なルールである。

思考プロセス: 段階的な修正

複数のファイルを一度に変更するような、大規模で複雑な計画を立てないこと。

まず1つのファイルを修正し、その変更が完了したことを確認してから、次のファイルの修正に移る、というように、タスクを小さなステップに分割して実行すること。

意図のコメント: コードを修正した場合、その意図や理由を説明するコメントをコード内に追加すること。これにより、人間がレビューする際の理解を助ける。

▼ 追加ルール: テストコードの同時生成 ▼

あなたが Code.gs または *Service.gs ファイル内の関数（ロギック）を変更、または新しく追加した場合、あなたは必ず Tests/UnitTests.gs ファイルも同時に変更しなければなりません。

テストの追加: 新しい機能を追加した場合、その機能に対する GasT.it(...) ブロックを Tests/UnitTests.gs に追加してください。

テストの修正: 既存の機能のバグを修正した場合、そのバグを検出できたはずのテストケースを追加・修正してください。

モックの使用: Google Apps Script のグローバルオブジェクト（DriveApp, SpreadsheetApp, PropertiesService など）は、Tests/UnitTests.gs にある mock(...) 関数を使用して、必ずモック（偽のオブジェクト）を作成してください。既存のテストが最良のサンプルです。

コミットメッセージのフォーマット

コミットメッセージは必ず英語で記述し、動詞から始めること (例: Fix: Handle null pointer exception in auth module)。

コミットは、Issueの番号（例: #123）で参照すること。

実行後の行動

コード修正後は、必ず新しい Pull Request を作成し、人間による最終レビューを要求すること。

コードの提出方法 (最重要ルール)

コードの修正が完了したら、あなた自身で git commit や git push を実行しようとしないでください。

代わりに、すべてのタスクが完了したことを示すために、以下の決まったフレーズだけを最終的な応答として出力してください。

All tasks are complete.

このフレーズを検知して、システムが自動的にコミットとPull Requestの作成を行います。

7. 特殊タスク: 新規モジュール生成

/new_module コマンドで起動された場合、あなたの役割は「コード修正」ではなく「機能モジュールの新規追加」です。

あなたは以下の2つの成果物を作成し、Pull Requestとして提出しなければなりません。

GASハンドラ関数の追加:

ユーザーの要望を実現する新しい関数を、適切な *Service.gs ファイル（例: SheetService.gs）の末尾に追加します。

関数は settings オブジェクトを引数に取ります。

モジュールJSONファイルの新規作成:

ModuleService.gs の createInitialModules にある定義を参考に、新しいモジュール（id, name, description, settings, handler 等）を定義したJSONファイルを作成します。

handler の値は、1で作成した関数名と一致させてください。

ファイルはリポジトリのルート（または /modules ディレクトリ）に配置してください。
